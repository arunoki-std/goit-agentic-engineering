# Домашнє завдання 2
 Conventions Extractor + власний ревʼювер агент зі скілами (API Contract Reviewer).

# Що робимо: 
Будуємо фічу, яка аналізує конвенції кодстайлу з репозиторію і пропонує їх перетворити на скіл.
Більшість знахідок дійсно можуть бути невалідними, але деякі з них реально можна допрацювати і перетворити у корисний скіл з правилами вашого проєкту. Додатковим завданням буде подумати як можна продуктово покращити цю фічу, щоб вона давала більше знахідок або ж кращої якості.

# Як юзер я можу:
- запустити аналіз репозиторію на Conventions
- побачити всі знайдені Conventions
- зробити accept/reject певного інсайту
- едитувати конкретний інсайт
- перейти на модал едитування скіла з інсайтів які я вибрав
- едитувати майбутній текст скіла з інсайтами та метадані
- зберегти скіл або ж відповитися від нього

# Можлива реалізація - Conventions Extractor:
1. Створіть таблицю conventions і роут POST /repos/:id/conventions/extract.
2. Реалізуйте відбір зразків — повністю кодом, без моделі: конфіги (eslint, tsconfig, prettier) + топ-12 файлів через готовий метод repoIntel.getConventionSamples().
3. Зробіть виклик дешевої моделі на аналіз репозиторію. Цей функціонал уже частково може бути зроблено. Модель має вертати: список кандидатів {категорія, правило, evidence: файл+рядок, впевненість}.
4. Додайте кодову перевірку доказів кандидатів у файлах проекту: чи файл існує? рядок з кодом існує? Кандидати без доказів відкидаються.
5. Зробіть UI-список кандидатів з кнопками approve / reject.
6. Зберіть approved-кандидати у один скіл repo-conventions і прилінкуйте його до агента (механізмом з лабораторної)
7. Також, як варіант, можна обернути цю фічу в ширшому напрямку і створювати багато скілів зі знахідок.

# Критерії приймання:
- Знято демо-відео, на якому видно, як працює фіча — Conventions Extractor у дії плюс експеримент із API Contract Reviewer (без скілів проти зі скілами);
- є відкритий Pull Request з гарним описом того, що було зроблено;
- Conventions Extractor дає результати на UI;
- з прийнятих кандидатів можна створити 1 скіл чи декілька; rejected-кандидати не потрапляють у підсумковий скіл;
- кожен кандидат на скіл має докази з реальним кодом, клік веде до файла з кодом на GitHub;
- згенерований скіл можна прилінкувати до агента та запускати на ревʼю;
- API Contract Reviewer зі скілами ловить breaking change, який без скілів пропускався.

# Як перевірити:
- Є відкрите демо-відео, на якому показано роботу фічі від запуску Extractor'а до прилінкованого скіла й експерименту з агентом;
- є відкритий Pull Request з гарним описом того, що було зроблено, і коротким звітом по якості знахідок Extractor'а;
- докази кандидатів клікабельні й ведуть на реальний код; скіл `repo-conventions` прилінкований до агента і запускається на ревʼю.

- підвищити якість знахідок або ж їх кількість (може бути складним, бо потрібно буде додавати нові дані у механізм repo-intel)
