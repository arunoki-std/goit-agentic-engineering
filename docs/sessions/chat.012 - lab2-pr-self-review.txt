     Plan: PR Self-Review Skill

     Context

     Before pushing a PR to GitHub, there is no automated local gate that checks
     code quality, architecture compliance, and security. This skill fills that
     gap — it orchestrates the existing react-best-practices,
     onion-architecture, code-review, and security-review skills across the
     changed files, classifies findings by severity, and blocks the user from
     opening a PR if any critical issue is found.

     The skill will be created using /skill-creator (not hand-written),
     following the project's established skill development workflow: scaffold →
     eval loop → description optimization.

     ---
     File Structure

     .claude/skills/pr-self-review/
     ├── SKILL.md                       ← main orchestration (≤500 lines)
     ├── agents/
     │   └── reviewer.md                ← subagent prompt template for parallel
     review passes
     └── references/
         └── severity-rules.md          ← classification rubric
     (critical/warning/info)
                                           with React-specific and
     onion-arch-specific examples

     No scripts needed for v1. skill-creator will produce the workspace at
     .claude/skills/pr-self-review-workspace/.

     ---
     Skill Design

     Trigger description

     Use before opening a GitHub pull request to run a local quality gate.
     Invoke this
     skill whenever the user runs or mentions `gh pr create`, `gh pr`, or says
     "open PR",
     "create PR", "push PR", "ready to merge", or "submit for review" — even if
     they do
     not explicitly ask for a review. Also invoke on the /pr-self-review
     command.
     Do NOT invoke during general code discussion or when reviewing someone
     else's PR.

     Allowed tools

     Bash(git:*, gh:*), Read, Skill

     Execution steps (in SKILL.md)

     1. Discover — git diff <default-branch>...HEAD --name-only
     Detect default branch dynamically: git remote show origin | grep 'HEAD 
     branch'
     2. Classify — split file list into:
       - Frontend: client/ prefix
       - Backend: server/ prefix
       - Other: root, docs, config, .claude/
     3. Run reviews (parallel where possible)
       - If frontend files present → invoke react-best-practices (arch check) +
     code-review --high scoped to client/ + security-review
       - If backend files present → invoke onion-architecture (arch check) +
     code-review --high scoped to server/ + security-review
       - Always → code-review --medium on the full diff
       - Frontend and backend passes run simultaneously (spawn both in same
     response before awaiting results)
     4. Collect & classify findings — read references/severity-rules.md;
     translate each finding into structured format:
     {file, issue, severity: critical|warning|info, suggested_fix}
     5. Gate decision
       - ≥1 critical → ❌ BLOCKED: N critical issue(s). Do NOT open PR until 
     resolved. + list all criticals
       - 0 criticals → ✅ PASSED: Ready to open PR + list warnings

     Key design choices

     ┌──────────────────┬───────────────────────────┬──────────────────────────┐
     │     Decision     │          Choice           │          Reason          │
     ├──────────────────┼───────────────────────────┼──────────────────────────┤
     │                  │                           │ Avoids embedding         │
     │ Subagents for    │ Yes, via                  │ sub-skill instructions   │
     │ parallel review  │ agents/reviewer.md        │ inline; mirrors          │
     │                  │                           │ skill-creator pattern    │
     ├──────────────────┼───────────────────────────┼──────────────────────────┤
     │ Severity         │ In SKILL.md +             │ Judgment-based, not      │
     │ classification   │ severity-rules.md         │ mechanical — belongs in  │
     │                  │ reference                 │ LLM instructions         │
     ├──────────────────┼───────────────────────────┼──────────────────────────┤
     │ Sub-skill        │ Via Skill tool (by name)  │ Avoids duplicating       │
     │ invocation       │                           │ instructions; stays thin │
     ├──────────────────┼───────────────────────────┼──────────────────────────┤
     │ Code-review file │ Context instruction in    │ code-review has no       │
     │  scoping         │ subagent prompt: "focus   │ native file filter param │
     │                  │ on files under client/"   │                          │
     ├──────────────────┼───────────────────────────┼──────────────────────────┤
     │ Architecture     │ agents/reviewer.md        │ These skills are         │
     │ skills output    │ translates prose output   │ Q&A-oriented, not        │
     │                  │ to structured format      │ list-oriented            │
     └──────────────────┴───────────────────────────┴──────────────────────────┘

     ---
     Implementation Sequence (using skill-creator)

     Phase 1 — Scaffold

     Invoke /skill-creator and provide:
     - Name: pr-self-review
     - Goal, trigger conditions, classification logic, orchestration steps,
     severity model, gate mechanism
     - Request: yes to subagents, yes to eval test cases

     Phase 2 — Eval test cases to feed skill-creator

     Test 1 — Critical found → must BLOCK

     ▎ "I'm about to run gh pr create. My branch adds a ReviewService that 
     ▎ imports directly from db/schema/repos in the service layer. Review the 
     ▎ PR."
     ▎ Assertions: output contains "BLOCKED", mentions the file path, classifies
     ▎ as "critical"

     Test 2 — No criticals → must PASS

     ▎ "I've moved a fetch() call from a component into useQuery in 
     ▎ src/lib/hooks/agents.ts. Run /pr-self-review before I open the PR."
     ▎ Assertions: output contains "PASSED", 0 criticals listed

     Test 3 — Mixed: frontend critical + backend warning

     ▎ "My branch changes client/ RepoCard (added useEffect with direct fetch())
     ▎ and server/ settings (added a constant). Ready to open a PR."
     ▎ Assertions: output is "BLOCKED", useEffect fetch() listed as critical, 
     ▎ constants addition not flagged as critical

     Phase 3 — Description optimization

     Run the trigger eval loop (20 queries including near-misses like gh pr 
     list, "review someone else's PR", "review last commit") to tune the
     description for trigger accuracy.

     ---
     Known Challenges

     1. code-review lacks a file filter — mitigate by prefacing Skill invocation
     with focus instructions in the subagent prompt
     2. Architecture skills produce prose, not lists — agents/reviewer.md
     instructs the subagent to translate to structured format before returning
     3. Default branch may not be main — detect dynamically with git remote show
     origin
     4. Parallel spawning requires discipline — SKILL.md must explicitly say
     "spawn BOTH subagents in the same response"

     Critical files to read before implementation

     - .claude/skills/skill-creator/SKILL.md — full skill-creator workflow
     - .claude/skills/skill-creator/references/schemas.md — skill schema and
     eval format
     - .claude/skills/react-best-practices/SKILL.md — what this skill can/cannot
     do
     - .claude/skills/onion-architecture/SKILL.md — what this skill can/cannot
     do
     - .claude/skills/skill-creator/agents/grader.md — grader pattern to mirror
     in reviewer.md
