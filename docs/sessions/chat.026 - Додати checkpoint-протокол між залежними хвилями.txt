     Plan: Checkpoint Protocol Between Dependent Waves

     Context

     The multi-agent pipeline divides feature work into waves: parallel
     implementers run in isolated worktrees that all start from the same HEAD.
     When Wave 2 tasks depend on Wave 1 output, those tasks fail or silently
     backport missing changes — because Wave 2 worktrees see the pre-Wave-1
     HEAD. The specs/agents-pipeline-changelist.md (P0 item "Додати
     checkpoint-протокол між залежними хвилями") formalises the fix: after a
     blocking wave the orchestrator must commit the merged results, creating a
     stable SHA; only then can dependent worktrees be launched.

     The change touches two agent definition files only. No new files are
     created.

     ---
     Acceptance Criteria (from spec)

     1. Planner explicitly marks waves that require a checkpoint.
     2. Dependent agents receive a baseline_sha.
     3. Implementer returns BLOCKED — never self-backports missing changes.

     Change 1 — planner.md

     Rule 6 (line 56) — add checkpoint marking requirement

     Current text ends at:

     ▎ Put shared contracts, barrels, migrations, generated artifacts, and 
     ▎ integration wiring in a single-owner step or a later integration wave.

     Append after that sentence:
     When a later wave depends on the output of an earlier one, mark the earlier
     wave `CHECKPOINT REQUIRED`. A checkpoint wave must include: (a) explicit
     test commands, (b) a commit message for the orchestrator to use, and (c)
     instruction to record the resulting SHA and pass it as `baseline_sha` to
     every dependent-wave agent. Do not schedule a dependent wave without a
     checkpoint between it and its blocking predecessor.

     ## Parallel Execution output template (lines 117-121) — extend format

     Replace the current template block:
     ## Parallel Execution
     ### Wave 1
     - Task A — owner scope, dependencies, expected handoff
     - Task B — owner scope, dependencies, expected handoff
     ### Integration Wave
     - Shared-file owner and integration checks

     With:
     ## Parallel Execution
     ### Wave 1 — CHECKPOINT REQUIRED
     - Task A — owner scope, dependencies, expected handoff
     - Task B — owner scope, dependencies, expected handoff

     **Checkpoint protocol (orchestrator runs before Wave 2):**
     1. Merge all Wave 1 worktrees into the feature branch.
     2. Run: `<test commands>`
     3. Commit: `git commit -m "chore: checkpoint — wave 1 complete"`
     4. Record the resulting SHA and pass it as `baseline_sha` to every Wave 2
     agent.

     ### Wave 2 (requires Wave 1 checkpoint; baseline_sha: <SHA recorded above>)
     - Task C — owner scope, dependencies, expected handoff

     ### Integration Wave (no worktree isolation — use `integrator` agent)
     - Shared-file owner and integration checks

     Waves with no dependent successors do not carry the CHECKPOINT REQUIRED
     marker.

     Change 2 — implementer.md

     Required Handoff list (lines 29-32) — add baseline_sha field

     After the existing four bullet points ("the target outcome...", "explicit
     owner scope...", "dependencies or outputs...", "a validation package..."),
     add:
     - `baseline_sha` — required when `Depends on` lists a prior wave; the SHA
     of that wave's checkpoint commit. Absent when the task has no upstream wave
     dependency.

     BLOCKED guard — upstream SHA check (after line 63)

     The current BLOCKED rule for missing upstream changes is on line 63:

     ▎ If the task depends on changes that are absent from that HEAD, return 
     ▎ BLOCKED instead of reconstructing them.

     Replace that sentence with:
     If the task packet includes `baseline_sha`, verify before editing that the
     checkpoint commit is in the worktree's ancestry:

       git merge-base --is-ancestor <baseline_sha> HEAD

     If that command exits non-zero, return `BLOCKED: baseline_sha <sha> is not
     an ancestor of worktree HEAD — orchestrator must merge the blocking wave
     and re-launch this agent`. Do not reconstruct or cherry-pick the missing
     changes.

     If no `baseline_sha` is provided but the task lists upstream wave
     dependencies, return `BLOCKED: task declares upstream wave dependency but
     baseline_sha was not supplied`.

     Handoff Report format — add Baseline field

     In the ## Required Handoff Report template (around line 92), after the ## 
     Status line, add a new line:

     ## Baseline
     <baseline_sha verified> | not applicable (no upstream wave dependency)

     ---
     Orchestration Rule (not in agent files — for operator/WORKFLOW.md 
     awareness)

     The orchestrator (main Claude session) is responsible for the steps the
     agents cannot do for themselves:

     1. Wait for all Wave N implementers to return COMPLETED.
     2. Run Wave N test commands on the feature branch.
     3. git commit -m "chore: checkpoint — wave N complete" on the feature
     branch.
     4. Copy the SHA (git rev-parse HEAD).
     5. Launch Wave N+1 agents, injecting baseline_sha: <SHA> into each task
     packet.

     This rule is implicit in the planner's checkpoint template and should also
     be noted in docs/lab-3/WORKFLOW.md when that document is next updated.

     ---
     Verification

     After applying the changes, launch a planner agent with a two-wave feature
     request and verify:

     1. The returned plan's ## Parallel Execution section shows CHECKPOINT 
     REQUIRED on Wave 1.
     2. Wave 2 tasks show baseline_sha: <placeholder> in the template.
     3. Launch an implementer with a task packet that omits baseline_sha but
     declares a Wave 1 dependency → it should return BLOCKED: baseline_sha was 
     not supplied.
     4. Launch an implementer with a mismatched baseline_sha (a fake SHA not in
     history) → it should return BLOCKED: baseline_sha … is not an ancestor of 
     worktree HEAD.
     5. Launch an implementer with the correct baseline_sha in ancestry → it
     should proceed normally and report ## Baseline: <sha> verified.
