  ---
  DevDigest — онбординг по проекту

  Ціль проекту

  DevDigest — локальний AI-інструмент для code review pull request'ів. Ти
  підключаєш GitHub-репозиторій, імпортуєш PR'и, налаштовуєш агентів (з різними
  моделями і промптами) — і вони роблять структурований review: знаходять
  проблеми з severity, score, і точними посиланнями на рядки в диффі.

  Це навчальний курс: стартовий шаблон вже повністю робочий (мінімальний
  end-to-end), а кожен урок додає одну фічу (cost badge, MCP-сервер, eval
  pipeline, multi-agent, тощо).

  ---
  Повний flow review (end-to-end)

  1. Add repo  →  server клонує git repo  →  repo-intel індексує
                 (AST grep + dep-cruiser = символи + граф залежностей)

  2. Import PRs  →  Octokit тягне diff/commits/body з GitHub
                 →  зберігає в Postgres

  3. Run Review  →  reviews/run-executor.ts збирає:
                 │   - diff (з БД або файлу)
                 │   - agent (model + system prompt + skills)
                 │   - repo map (з repo-intel)
                 ▼
                 reviewer-core/review/run.ts
                 │   - вибирає стратегію: single-pass або map-reduce
                 │     (map-reduce = великий diff → по файлу окремо)
                 │   - assemblePrompt() → messages
                 │   - llm.completeStructured<Review>()
                 │   - reduceReviews() (якщо map-reduce)
                 ▼
                 grounding gate
                 │   - перевіряє кожен finding проти реального диффу
                 │   - дропає "галюцинації" (знаходження з неіснуючими рядками)
                 ▼
                 persists findings в Postgres
                 streams events через SSE → client live log

  ---
  Дивні / важливі нюанси

  1. Без monorepo workspace — пакети зв'язані tsconfig aliases, не npm. Зміна в
  reviewer-core/ одразу видна серверу без rebuild.
  2. Міграції не запускаються автоматично — якщо запустив сервер без pnpm 
  db:migrate, отримаєш relation does not exist. Завжди треба запускати вручну.
  3. Grounding gate — обов'язковий механічний фільтр після LLM. Якщо модель
  вигадала номер рядка, finding дропається. Це запобігає "тихим" галюцинаціям.
  4. pgvector — БД не просто Postgres. Використовується векторний пошук (для
  repo-intel / embeddings). Розширення активується першою міграцією
  0000_init.sql.
  5. SSE замість WebSocket — live log під час review стримується через
  Server-Sent Events (fastify-sse-v2).
  6. LLM провайдер інжектується — reviewer-core не знає, OpenAI чи Anthropic. Це
  дає mock-тестування без реальних API-викликів.
  7. E2E без LLM — браузерні тести детерміновані (JSON-флоу через
  agent-browser), жодних реальних LLM-запитів.

  засідить демо-дані, запустить API (:3001) і фронт (:3000).

❯ чи варто між задачами додавати команду /clear ? вона очищає консоль чи ще щось

/clear очищає контекст сесії — не тільки екран, а всю історію розмови з пам'яті моделі. Після /clear я починаю "з нуля" і не пам'ятаю нічого з попередньої розмови.

---
Коли варто використовувати

Так — між непов'язаними задачами:
- Закінчив роботу з server/, переходиш до зовсім іншої теми в client/
- Нова задача не потребує контексту попередньої
- Хочеш зекономити токени (бачиш що контекст вже 60-70%+)

Ні — коли задачі пов'язані:
- Продовжуєш роботу в тому ж модулі
- Нова задача спирається на рішення з попередньої
- Хочеш щоб я пам'ятав що вже зробили

---
Практична порада для твого сценарію

Твій поточний flow міг би виглядати так:

задача 1  →  /export  →  /clear  →  задача 2  →  /export  →  /clear ...

При цьому CLAUDE.md і LEARNINGS.md компенсують /clear — наступна сесія підхоплює контекст проекту автоматично з файлів, а не з пам'яті.

Це власне і є задумка системи яку ми налаштували: стан зберігається у файлах, а не в контексті.
