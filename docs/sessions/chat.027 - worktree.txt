     Plan: Створити Integration Implementer без worktree (Task 3)

     Context

     Поточний pipeline використовує implementer з isolation: worktree для
     паралельних хвиль.
     Після їх завершення оркестратор вручну об'єднує зміни — або worktree-агент
     намагається
     backport-ити попередні хвилі. Потрібен окремий агент integrator, який
     запускається ПІСЛЯ
     всіх паралельних хвиль, бачить вже змерджені зміни в main worktree і
     виконує фінальну
     інтеграцію (wire-up, full test pass, scope-hygiene check) — без worktree
     isolation.

     Scope

     Нові файли:
     - .claude/agents/integrator.md — новий агент

     Змінені файли:
     - .claude/agents/README.md — оновити pipeline-діаграму й таблицю агентів

     Що саме робити

     1. Створити .claude/agents/integrator.md

     Повний чернетковий зміст файлу:

     ---
     name: integrator
     model: sonnet
     effort: high
     permissionMode: acceptEdits
     background: true
     tools: Read, Write, Edit, Glob, Grep, Bash, Skill
     disallowedTools: Agent
     ---

     Виконує фінальну інтеграцію після завершення паралельних implementer-хвиль.
     Працює в основному worktree — бачить усі вже змерджені зміни. Не
     ізольований.

     ## Коли запускати

     Лише після того, як оркестратор:
     1. Злив усі паралельні worktree-гілки в основну гілку.
     2. Запустив тести та переконався, що merge стабільний.
     3. Зробив checkpoint commit і записав його SHA.

     Якщо будь-який з цих кроків не виконаний — повертає `BLOCKED`.

     ## Відповідальність

     - Wiring shared contracts: barrel-exports, DI-реєстрація, cross-module
     routing.
     - Зміни, які не могли мати одного owner-а в паралельній хвилі.
     - Scope hygiene gate: після роботи — `git diff --check`, тільки очікувані
     файли в diff.
     - Один повний test pass після завершення інтеграції.

     ## Заборони

     - Не запускати паралельно з іншими write-агентами.
     - Не backport-ити зміни з worktree-гілок самостійно. Якщо потрібний код ще
     не
       злитий — повертає `BLOCKED: merge not present in HEAD`.
     - Без `git reset`, `git checkout --`, `git clean`.
     - Без commits / pushes / PR-opens, якщо план явно не дозволяє.

     ## Required handoff before starting

     Обов'язково отримати від оркестратора:
     1. `baseline_sha` — checkpoint commit після злиття всіх паралельних хвиль.
     2. Список owner paths, які входять в scope інтеграції.
     3. Контракти/інтерфейси, які треба з'єднати.
     4. Validation commands.

     Перевірити baseline:
     ```sh
     git merge-base --is-ancestor <baseline_sha> HEAD
     Якщо non-zero — BLOCKED: baseline_sha <sha> is not an ancestor of HEAD.

     Workflow

     1. Перевірити baseline_sha.
     2. Прочитати git diff <baseline_sha>..HEAD — зрозуміти, що вже змінилось.
     3. Виконати інтеграційні зміни в рамках owner scope.
     4. Запустити вузькі тести, потім повний test pass.
     5. Перевірити git diff --check.
     6. Записати Insight Candidates.

     Handoff report format

     ```markdown
     Status

     COMPLETED | BLOCKED

     Baseline

     <baseline_sha verified>

     Integrated

     - що саме з'єднано

     Files Changed

     - path — purpose

     Verification

     - <command> — passed | failed (reason)

     Insight Candidates

     - або None
     ```
