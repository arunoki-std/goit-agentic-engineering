     Plan: "Create skill from conventions" modal

     Context

     The Conventions page surfaces extracted house-rules backed by code
     evidence. Users accept individual candidates, then want to package all
     accepted ones into a reusable Skill for the Skills Lab. The backend already
     has POST /repos/:id/conventions/skill-preview and POST 
     /repos/:id/conventions/skill. The frontend needs a modal that calls preview
     on open, lets users edit the generated skill, then calls create on submit.

     Mockup reference: specs/convention-extractor-homework/mockups/convensions -
     create skill screen.png

     ---
     Files to create

     1. client/src/app/repos/[repoId]/conventions/_components/CreateSkillModal/C
     reateSkillModal.tsx

     Main modal component. Key behaviours:

     - On mount (useEffect): call
     useConventionSkillPreview(repoId).mutate(undefined, { onSuccess }) —
     populates form fields from API response.
     - Loading state: show <Skeleton> blocks while preview is pending
     (preview.isPending).
     - Info banner: "Merged from {acceptedCount} accepted conventions in 
     {repoName}. Everything below is editable before you save." — uses
     accent-coloured info box pattern.
     - Editable fields:
       - Name (TextInput, required)
       - Description (TextInput)
       - Type (SelectInput, options: rubric/convention/security/custom, default:
     "convention")
       - Enabled toggle (Toggle from @devdigest/ui primitives, default: true) +
     hint label
       - Skill body (raw textarea with editor chrome — filename header + token
     count, same pattern as ConfigTab.tsx:82–148)
     - Token count: preview.data.token_count when body unchanged;
     Math.ceil(body.length / 4) when body is edited — mirrors
     ConfigTab.tsx:15–17.
     - Footer:
       - Left: "→ Saved as v1 · added to Skills Lab" hint (muted text)
       - Right: Cancel (secondary) + Create skill (primary, disabled when name
     or body empty or create.isPending)
     - On Create: create.mutateAsync(...), then toast.success(...), then call
     onClose(). useCreateConventionSkill already invalidates ["skills"] and
     ["conventions", repoId] on success.
     - On Cancel: call onClose() without any API call.

     Props:
     interface Props {
       repoId: string;
       repoName: string;    // passed from ConventionsView via useActiveRepo()
       acceptedCount: number;
       onClose: () => void;
     }

⏺ Two candidates pass the filter:

  1. useConventionSkillPreview is a mutation, not a query — semantically a read,
  but POST-backed and imperative, so callers must trigger it via useEffect on
  mount. Non-obvious for anyone reaching for useQuery.
  2. Mocking mutate as vi.fn() deliberately keeps form empty — when mutate(data,
  { onSuccess }) is mocked with a no-op, the onSuccess callback is never
  called, leaving state at initial values. Useful for testing disabled-button
  states without simulating API responses.
