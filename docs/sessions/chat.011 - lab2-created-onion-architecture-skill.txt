     Plan: Onion Architecture Skill for Backend Modules

     Context

     The server/ backend already has solid layering (routes → service →
     repository → DB), but there are no written rules enforcing Onion
     Architecture boundaries. Without a skill, developers can accidentally bleed
     Drizzle imports into services, put business logic in routes, or let
     modules call each other's repositories directly. This skill gives Claude an
     authoritative, project-grounded guide to answer "where does this code go?"
     questions for the backend — mirroring what react-best-practices does for
     the frontend.

     Skill: Content Plan

     Frontmatter

     name: onion-architecture
     description: Answer architecture and file-placement questions for the
     server/ backend — which layer owns this logic, what can import what, where
     repository interfaces go, how the DI container is the composition root, and
     how to keep Drizzle out of services. Use whenever the user asks about
     backend module structure, cross-module dependencies,
     service/repository/adapter boundaries, domain model placement, or "where 
     does this code go?" in server/.
     allowed-tools: Read, Bash(grep:*, find:*)

     Skill Body Sections

     1. Layer Map (this project's files → Onion rings)

     Domain Core        vendor/shared/contracts/ + domain types in
     modules/<name>/types.ts
     Domain Services    modules/<name>/service.ts  (pure business logic)
     App Services       modules/<name>/service.ts  (orchestration: job enqueue,
     cross-adapter calls)
     Infrastructure     adapters/<service>/  · db/  · platform/
     Presentation       modules/<name>/routes.ts  (HTTP transport only)
     Composition Root   platform/container.ts

     2. The Golden Rule: Dependency Direction

     - Inner rings NEVER import outer rings
     - Routes → Service → Repository Interface → (Container provides impl)
     - Services NEVER import db/schema/* directly — only via repository methods
     - Adapters implement interfaces from vendor/shared/adapters.ts — services
     depend on the interface

     Forbidden imports table:

     ┌───────────────┬────────────────────────────────────────────────────────┐
     │     Layer     │                     May NOT import                     │
     ├───────────────┼────────────────────────────────────────────────────────┤
     │ service.ts    │ db/schema/*, routes.ts, adapters/<name>/ (directly)    │
     ├───────────────┼────────────────────────────────────────────────────────┤
     │ routes.ts     │ db/, adapters/ (directly), another module's service.ts │
     ├───────────────┼────────────────────────────────────────────────────────┤
     │ repository.ts │ platform/jobs.ts, adapters/, other modules             │
     ├───────────────┼────────────────────────────────────────────────────────┤
     │ adapters/<x>/ │ modules/, db/schema/*                                  │
     └───────────────┴────────────────────────────────────────────────────────┘

     3. Ports & Adapters in this Project

     - Port (interface) = vendor/shared/adapters.ts for cross-cutting concerns,
     or modules/<name>/types.ts for module-scoped repo interface
     - Adapter (implementation) = adapters/<service>/index.ts or
     modules/<name>/repository.ts
     - Container = composition root that wires ports → adapters — services get
     deps via container.*, not direct imports

     4. Cross-Module Communication Rules

     - Module A must NEVER call Module B's repository.ts directly
     - Module A can call Module B's service.ts (via container if injectable, or
     direct instantiation)
     - Preferred: extract a shared contract to vendor/shared/ and inject via
     container

     5. Repository Pattern — Correct Form

     // ✅ Service depends on interface, not Drizzle
     interface RepoRepository {
       findByFullName(workspaceId: string, fullName: string): Promise<Repo |
     null>;
     }
     class RepoService {
       constructor(private container: Container) {}
       async add(workspaceId: string, url: string) {
         const repo = await new RepoRepository(this.container.db)
           .findByFullName(workspaceId, parsedName);
         // ← repo layer handles Drizzle, service just calls methods
       }
     }

     // ❌ Drizzle leaking into service
     import { repos } from '../../db/schema/repos';
     class RepoService {
       async add(...) {
         await db.select().from(repos).where(eq(repos.workspaceId, id)); // ← 
     belongs in repository
       }
     }

     6. Checklist for a New Module

     - [ ] Define domain types in modules/<name>/types.ts
     - [ ] Write repository interface before implementation
     - [ ] routes.ts does: parse input, call service, return DTO — nothing else
     - [ ] service.ts does: business rules, adapter calls via container, enqueue
     jobs
     - [ ] repository.ts does: Drizzle queries only, always scoped by
     workspaceId
     - [ ] helpers.ts: pure DTO converters, no side effects
     - [ ] Register in modules/index.ts
     - [ ] Register job handlers in platform/jobs.ts if background work needed

     7. Testing Alignment with Onion Architecture

     - Services are unit-testable by providing ContainerOverrides (mock
     adapters)
     - Repositories must hit a real DB via @testcontainers/postgresql (no
     mocking the DB layer)
     - Routes tested via Fastify inject() with a real container or overrides
     - Never mock a repository in a service test — use ContainerOverrides to
     swap the whole adapter

     8. Anti-Patterns to Flag

     - import { repos } from '../../db/schema' inside service.ts → move to
     repository.ts
     - Business logic (if, computation, validation beyond Zod) in routes.ts →
     move to service.ts
     - import { ReviewService } from '../reviews/service' directly in another
     module → go through container or shared interface
     - repository.ts calling container.jobs.enqueue() → job scheduling belongs
     in service.ts
     - Raw db.query() in service.ts or routes.ts → route to repository

     9. Quick-Reference Cheat Sheet

     modules/<name>/routes.ts      ← HTTP only: parse → call service → return
     DTO
     modules/<name>/service.ts     ← Business logic, adapter calls via container
     modules/<name>/repository.ts  ← Drizzle queries, workspace-scoped, one
     table
     modules/<name>/types.ts       ← Domain types, repository interface
     modules/<name>/helpers.ts     ← Pure DTO transforms
     modules/<name>/constants.ts   ← Literals, job kind strings

     platform/container.ts         ← Composition root; inject here, not in
     service files
     adapters/<service>/           ← Outer ring; implement vendor/shared
     interfaces
     vendor/shared/adapters.ts     ← Port interfaces (SecretsProvider, etc.)
     vendor/shared/contracts/      ← Zod schemas for API + internal data shapes
     db/schema/<domain>.ts         ← Drizzle table definitions; only imported by
     repository.ts

     ---
     My Suggestions to Add (beyond the standard template)

     1. "Sliced Onion" note: The project already uses vertical slices (modules
     by feature), not horizontal layers. The skill should explain that this is
     intentional — the Onion rings are within each slice, not across the
     codebase. Reference Oliver Drotbohm's "Sliced Onion Architecture."
     2. Domain Events placeholder: When a service needs to notify another module
     (e.g., after a repo is added, trigger indexing), the current approach is
     direct job enqueueing. The skill should note where domain events would fit
     when the project grows.
     3. Drizzle-specific guidance: Drizzle is a query builder, not full
     ActiveRecord — so "keeping Drizzle out of services" is more nuanced than
     with Prisma. The skill should explain that db/schema/* imports are the red
     line (not the db client itself, which can flow through container).
     4. Inter-module dependency graph: A simple text diagram showing which
     modules currently depend on which, so developers know the permitted call
     graph.
     5. When NOT to add a layer: Anti-advice — for a simple CRUD module with no
     business logic, a thin service is fine; don't force domain events or rich
     domain models where they add no value.

     Verification

     After implementation:
     1. Ask Claude: "Where should I put a function that validates a GitHub URL
     format before saving a repo?" — expect it to say modules/repos/helpers.ts
     (pure transform) or service.ts if it throws a domain error.
     2. Ask: "Can my ReviewService import Drizzle schema directly?" — expect a
     clear NO with an explanation to route through repository.ts.
     3. Ask: "A new module notifications needs to know when a review finishes —
     how?" — expect guidance about calling NotificationService from
     ReviewService via container, or a job-based decoupling.
     4. Check that the trigger fires correctly: ask an architecture question
     about server/ without saying "onion architecture."
