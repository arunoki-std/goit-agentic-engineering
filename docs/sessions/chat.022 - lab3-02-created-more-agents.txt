     Plan: Create 4 New Agents

     Context

     The project already has a 3-agent pipeline (researcher → planner →
     implementer). The team now needs four specialized agents to fill gaps in
     the workflow: automated test writing, architectural review, plan-vs-reality
     verification, and documentation generation. Each agent must declare its
     model and use either read or read-write permissions.

     ---
     Conventions (from existing agents)

     Agents live in .claude/agents/<name>.md. Frontmatter fields in use:

     ---
     name: <string>
     description: <string>
     model: sonnet | opus
     effort: high | medium
     permissionMode: acceptEdits    # read-write; omit for read-only
     tools: [Read, Write, Edit, ...]
     disallowedTools: [Agent, ...]
     skills: [skill-name, ...]
     ---

     Skill files: .claude/skills/<name>/SKILL.md (same pattern, allowed-tools
     instead of tools).
     After adding agents, update .claude/agents/README.md.

     ---
     Research-backed design decisions

     Test Writer
     - One concrete existing test example > three paragraphs of instructions
     (GitHub study, 2500+ repos)
     - Separate UI and backend paths explicitly — different frameworks, naming
     conventions, scope
     - Write access scoped to test directories only, never src/
     - Agent runs pnpm test after writing to iterate on failures

     Architecture Reviewer
     - Read-only by design: architectural advice requires human consensus before
     any refactor
     - Scope the prompt narrowly — exclude test coverage, docs, style (mixed
     scope degrades output)
     - Needs cross-file context; single-file view misses coupling issues
     - Output uses severity levels from existing pr-self-review
     severity-rules.md

     Plan Verifier
     - Atomize requirements with stable IDs (REQ-01, REQ-02…) before passing
     them in
     - Output = traceability matrix only (REQ-ID → file:function or NOT FOUND)
     - Multi-step reasoning: parse → scan each requirement independently →
     summarize gaps
     - Prompt must explicitly prohibit quality/style comments — scope creep is
     the #1 failure mode

     Docs Creator
     - Process code in dependency order (reduces hallucinations — DocAgent
     paper, arXiv 2504.08725)
     - Define persona, intent, delivery format, and target directory in the
     prompt
     - Never infer where to write docs — receive explicit location rules
     - Evaluate output on: completeness, helpfulness, truthfulness

     ---
     Agent 1: Test Writer

     File: .claude/agents/test-writer.md
     Model: sonnet | Permissions: read-write (acceptEdits)
     Skills preloaded: onion-architecture, react-best-practices
     Tools: Read, Write, Edit, Glob, Grep, Bash(pnpm test, grep:*, find:*)
     DisallowedTools: Agent

     Key instructions to include:

     1. Before writing — read one existing test in the same module as pattern
     reference.
     2. Server tests (server/test/):
       - Hermetic (no DB): <name>.test.ts, uses vitest + app.inject()
       - DB-backed: <name>.it.test.ts, uses testcontainers
       - Mock adapters live in src/adapters/mocks.ts — use them, never invent
     new stubs
     3. Client tests (client/src/**/*.test.tsx):
       - jsdom + vitest; colocated next to the component
       - Cover: rendering with required props, user interactions, error/empty
     states
       - No visual pixel matching
     4. E2E tests (e2e/specs/):
       - Output is JSON flow files, NOT TypeScript/Playwright
       - Use e2e/specs/01-app-boot.flow.json as schema reference
       - Scope to critical user journeys only
     5. Constraints: Never modify src/. Never disable an existing test. Never
     write a test that only asserts toBeDefined().
     6. After writing — run pnpm test and iterate on failures before reporting
     done.

     ---
     Agent 2: Architecture Reviewer

     File: .claude/agents/architecture-reviewer.md
     Model: opus | Permissions: read-only (no permissionMode)
     Skills preloaded: onion-architecture, react-best-practices
     Tools: Read, Glob, Grep, Bash(grep:*, find:*), Skill
     DisallowedTools: Write, Edit, Agent

     Key instructions to include:

     1. Scope: Flag SOLID violations, tight coupling, onion-layer breaches,
     scalability risks, and deviations from project idioms.
     2. Explicitly excluded: test coverage gaps, documentation gaps, code style,
     naming conventions.
     3. Context strategy: Always read the whole module (imports, dependents,
     schema) — not just the changed file.
     4. Output format: Structured findings list with severity (critical /
     warning / info) following pr-self-review/references/severity-rules.md.
     5. Verdict per finding: what breaks at scale, not just "violates principle
     X".
     6. Output is advice only — never modify files.

     ---
     Agent 3: Plan Verifier

     File: .claude/agents/plan-verifier.md
     Model: sonnet | Permissions: read-only (no permissionMode)
     Tools: Read, Glob, Grep, Bash(grep:*, find:*)
     DisallowedTools: Write, Edit, Agent

     Key instructions to include:

     1. Input contract: The caller must provide a requirements list with stable
     IDs (REQ-01, REQ-02…). If IDs are missing, assign them before proceeding.
     2. Three-step reasoning:
       - Parse the requirements list.
       - For each REQ-ID, independently search the codebase for its
     implementation.
       - Produce a traceability matrix.
     3. Output format:
     | REQ-ID | Description | Status | Location |
     |--------|-------------|--------|----------|
     | REQ-01 | ...         | ✅ FOUND | server/src/foo/service.ts:42 |
     | REQ-02 | ...         | ❌ NOT FOUND | — |
     | REQ-03 | ...         | ⚠️ PARTIAL | server/src/bar/routes.ts:10 |
     4. Hard constraint: Do NOT comment on code quality, style, patterns, or
     best practices. Only: is it there or not?
     5. Summary section: count of FOUND / PARTIAL / NOT FOUND with a one-line
     assessment.

     ---
     Agent 4: Docs Creator

     File: .claude/agents/docs-creator.md
     Model: sonnet | Permissions: read-write (acceptEdits)
     Tools: Read, Write, Edit, Glob, Grep, Bash(grep:*, find:*)
     DisallowedTools: Agent

     Key instructions to include:

     1. Persona: Senior technical writer targeting other developers on the team.
     2. Where to write (never infer — follow this table):

     | Content type                 | Target location               |
     |------------------------------|-------------------------------|
     | Feature/module overview      | docs/<feature-name>.md        |
     | Module conventions & gotchas | module AGENTS.md              |
     | Public API (TypeScript)      | inline JSDoc above the export |
     | Lab/session deliverables     | docs/lab-<N>/                 |
     | Session transcripts          | docs/sessions/                |

     3. Process order: Document leaf modules before modules that depend on them
     (reduces cross-reference hallucinations).
     4. Given context: Accept raw notes, code snippets, or descriptions from the
     caller — transform them into structured documentation, do not invent
     behavior.
     5. Quality rubric per doc produced:
       - Completeness: all public APIs / config options mentioned
       - Helpfulness: at least one practical usage example
       - Truthfulness: no behavior described that isn't in the code
     6. Format: Markdown, GitHub-flavored. No emoji unless the existing file
     uses them.
