     Plan: Тестування пайплайну агентів DevDigest

     Context

     Потрібно спостерігати пайплайн агентів DevDigest у дії — побачити кожен
     крок виконання, виявити слабкі та сильні сторони. Пайплайн складається з
     чотирьох засіяних агентів (General, Security, Performance, Test Quality),
     що запускаються паралельно на diff PR і проходять через: вибір стратегії →
     складання промпту → LLM-виклик → злиття результатів (map-reduce) → gate
     grounding (видаляє "галюциновані" посилання).

     Що саме тестуємо

     diff + repo map → assemblePrompt() → LLM → reduceReviews() →
     groundFindings() → findings

     Ключові файли:
     - reviewer-core/src/review/run.ts — reviewPullRequest(), вхідна точка
     движка
     - reviewer-core/src/grounding.ts — citation gate
     - server/src/modules/reviews/run-executor.ts — оркестрація, фоновий запуск
     - server/src/modules/reviews/service.ts — resolves агентів, стартує runs
     - server/src/db/seed.ts / seed-prompts.ts — 4 засіяні агенти

     Кроки тестування

     1. Перевірка середовища

     # Перевірити секрети
     cat ~/.devdigest/secrets.json

     # Запустити стек
     ./scripts/dev.sh

     Перевірити, що API (:3001) і web (:3000) підняті. Переконатися, що є хоча б
     один LLM API key.

     2. Огляд засіяних даних

     # Список агентів
     curl -s http://localhost:3001/agents | jq '.[] | {id, name, provider, 
     model, strategy}'

     # Список репозиторіїв (seeded: acme/payments-api)
     curl -s http://localhost:3001/repos | jq '.[0] | {id, name, indexed}'

     # PRs у репозиторії
     REPO_ID=$(curl -s http://localhost:3001/repos | jq -r '.[0].id')
     curl -s "http://localhost:3001/pulls?repoId=$REPO_ID" | jq '.[] | {id, 
     number, title}'

     3. Сценарій A — Happy Path: всі 4 агенти на одному PR

     # Запустити всі агенти одночасно
     PR_ID=<один із засіяних PR>
     curl -s -X POST "http://localhost:3001/pulls/$PR_ID/reviews" \
       -H "Content-Type: application/json" \
       -d '{"all":true}' | jq '{runs: [.runs[] | {run_id, agent_name}]}'

     Для кожного run_id — відкрити SSE-стрим в окремому терміналі:
     curl -N "http://localhost:3001/runs/$RUN_ID/events"

     Що спостерігати:
     - Послідовність подій: info → tool → result
     - Повідомлення: "Reviewing N changed file(s) in one pass" vs "Large diff → 
     map-reduce over N files"
     - "X candidate finding(s)" після LLM-відповіді
     - "Citation grounding: N/M passed" — скільки знахідок видалено

     4. Сценарій B — Аналіз результатів після завершення

     # Статус усіх ранів
     curl -s "http://localhost:3001/pulls/$PR_ID/reviews" | jq \
       '.runs[] | {agent: .agentName, status, score, findings: .findingsCount, 
     grounding, duration_ms}'

     Що порівняти:
     - score кожного агента на одному PR — чи розходяться оцінки?
     - grounding ("N/M passed") — який % знахідок виживає?
     - findingsCount — хто знаходить більше?

     5. Сценарій C — Trace одного рану (детальний)

     RUN_ID=<один із run_id>
     curl -s "http://localhost:3001/runs/$RUN_ID" | jq '{status, provider, 
     model, duration_ms, tokens_in, tokens_out, score, blockers, grounding}'

     Якщо є доступ до БД:
     psql $DATABASE_URL -c "SELECT run_id, (trace->>'events')::jsonb FROM 
     run_traces WHERE run_id = '$RUN_ID';"

     6. Сценарій D — Перевірка Test Quality Reviewer (зі скілами)

     # Знайти ID Test Quality Reviewer
     AGENT_ID=$(curl -s http://localhost:3001/agents | jq -r '.[] | select(.name
     | contains("Test Quality")) | .id')

     # Переконатися, що скіли підключені
     curl -s "http://localhost:3001/agents/$AGENT_ID/skills" | jq '.[] | {name, 
     order}'

     # Запустити тільки цей агент
     curl -s -X POST "http://localhost:3001/pulls/$PR_ID/reviews" \
       -H "Content-Type: application/json" \
       -d "{\"agent_id\": \"$AGENT_ID\"}" | jq

     7. Сценарій E — Версіонування конфігурації

     # Змінити системний промпт агента
     curl -s -X PUT "http://localhost:3001/agents/$AGENT_ID" \
       -H "Content-Type: application/json" \
       -d '{"systemPrompt": "Test prompt change to trigger version bump"}' | jq
     '.version'

     # Переглянути версії
     curl -s "http://localhost:3001/agents/$AGENT_ID/versions" | jq '.[] | 
     {version, createdAt}'

     Сильні сторони (передбачувані)

     - Паралельний запуск агентів з ізольованими збоями
     - Grounding gate захищає від "галюцинацій" у цитатах
     - SSE-стримінг дає реал-тайм спостереження

     - Grounding gate перевіряє лише рядкові посилання, не семантику знахідок

  ---
  Сильні сторони

  1. Grounding gate захищає від галюцинацій: 100% для single-pass, і коректно
  блокує неправдиві посилання для map-reduce
  2. SSE-стримінг реал-тайм: кожна подія (tool → result) з'являється миттєво, не
  батчем
  3. Fallback diff: якщо git diff не доступний — відновлюється з pr_files (з
  патчами)
  4. Repo-intel кешується: repo map: cached=true — без зайвих токенів при
  повторних ревʼю
  5. Ізоляція збоїв: кожен агент незалежний, fallback на рівні рану
  6. Версіонування: system_prompt → version bump 1→2, snapshot зберігається

  Слабкі сторони

  1. Single-pass на великому diff = 208K токенів / 4.3 хв: без auto-стратегії
  агент відправляє весь diff одним запитом — дорого і повільно
  2. Map-reduce без cross-file context: агент бачить файли ізольовано,
  "знаходить" зміни у файлах поза diff → grounding скидає всі знахідки
  3. Нерелевантні файли в map-reduce: tsconfig.tsbuildinfo (бінарний),
  docs/sessions/*.txt (логи сесій) потрапляють у diff і витрачають ~30с кожен на
  LLM-виклик
  4. Demo PRs дають score=100 без реального ревʼю: head_sha: "demo" → 0 файлів →
  всі агенти approve з 0 знахідок — тихий false positive
  5. Score divergence: LLM звітує 89, система перераховує до 85 — добре, але цей
  gap означає, що модель переоцінює себе

  Несподіванки

  - agent_name: null у відповіді POST /review (поле не заповнюється в initial
  response)
  - GET /runs/:id не існує як маршрут — дані тільки через /runs/:id/trace
