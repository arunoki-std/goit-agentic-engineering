  Готово. Ось що підготовлено:

  Демо сценарій по кроках

  Part 1 — Conventions Extractor (2.5 хв):
  1. Open DevDigest → repo → Conventions → Scan → candidates з'являються
  2. Accept 3 кандидати, reject 1 → клікнути GitHub evidence link (доводить
  реальний код)
  3. Create skill → modal показує тільки accepted → підтвердити що rejected
  відсутній → зберегти

  Part 2 — API Contract Reviewer (2.5 хв):
  4. Відкрити Agents → "API Contract Reviewer" (без linked skill) → Run review
  на demo PR
  5. Результат: 0 CRITICAL, агент каже "naming refactor" — breaking change
  пропускається
  6. Skills → додати API Contract Reviewer Rules skill → Agents → прилінкувати
  7. Run review на той самий PR → 3–4 CRITICAL знахідки: transaction_id rename,
  account_number deleted, status code change

  Який breaking change використаний

  Файл fixtures/demo-payments-api.ts вводить 4 порушення в payments API:
  - transaction_id → transactionId (rename без alias)
  - account_number — видалено без deprecation
  - created_at → createdAt (rename без alias)
  - POST status 201 → 200 (зміна status code)

  Без skill — LLM бачить "camelCase refactor", не flagує breaking change.
  Зі skill — skill явно задає: CRITICAL за rename без alias, CRITICAL за
  видалення поля, CRITICAL за зміну status code → знахідки стають
  передбачуваними і специфічними.

  Що саме показати у відео

  Ключовий контраст — side-by-side двох review runs на одному PR: пустий список
  CRITICAL без skill vs 3–4 CRITICAL знахідки зі skill. Також обов'язково клік
  на GitHub evidence link під час Conventions flow — доводить що кожне правило
  прив'язане до реального коду.

  Всі деталі в DEMO.md, skill body в fixtures/api-contract-reviewer-skill.md, PR
  draft в PR_DESCRIPTION.md.

⏺ Two non-obvious insights from this session:

  1. What Works — skills are injected before the diff in the prompt (line 109 of
  prompt.ts), so the model reads explicit rules before it reads the code. This
  is why linked skills make findings consistent and specific rather than
  probabilistic.
  2. What Doesn't Work — demo PRs that modify existing files: only the changed
  lines are groundable by the citation gate. A breaking change that sits on an
  unchanged line gets dropped as hallucinated. Using a new fixture file (all
  lines = additions) sidesteps this silently.
